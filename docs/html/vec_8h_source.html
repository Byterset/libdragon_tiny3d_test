8 // A fast, generic, contiguous growable array type.
9 //
10 // In memory, it looks like this:
11 //
12 // len capacity elem_size data
13 // +---------+----------+---------+---------+
14 // | 2 | 3 | 4 | 0x06577 |
15 // +---------+----------+---------+---------+
16 // _____________________________|
17 // |
18 // V
19 // Heap
20 // +------+------+------+
21 // | 42 | 69 | |
22 // +------+------+------+
23 //
24 //
25 // # Indexing
26 // The `vec_t` type allows to access values by index.
27 //
28 // One must be careful when mutating the vector like this, as indexing outside
29 // of the allocated memory is undefined behavior (UB). Always cast the
30 // underlying data to the type you have declared it with.
31 // The library provides the "safer" `VEC_MUT()` macro for this operation.
32 //
33 // An example: ```c
34 // // Declare a new `vec_t` of size 2, storing integers
35 // vec_t* v = vec_with_capacity(2, sizeof(int));
36 //
37 // // Set the value of the first element to 42
38 // ((int*)v->data)[0] = 42;
39 // // Same thing with `VEC_MUT()`
40 // int x = 42;
41 // VEC_MUT(v, int, 0, x);
42 // ```
43 //
44 //
45 // # Capacity and reallocation
46 // The capacity of a vector is the amount of space allocated for any future
47 // elements that will be pushed/inserted onto the vector. This is not to be
48 // confused with the length of a vector, which specifies the number of actual
49 // elements within it. If a vector’s length exceeds its capacity, it will
50 // automatically be increased, but its elements will have to be reallocated.
51 //
52 // To keep going with our example:```c
53 // // Pushing onto our vector doesn't need a reallocation because we declared
54 // // it with a size of 2 (i.e. `v->capacity = 2`)
55 // vec_push(v, 69);
56 //
57 // // Pushing again will cause a reallocation!
58 // vec_push(v, 420);
59 // ```
60 //
61 // Reallocating memory can be slow, especially on large vectors. For this reason,
62 // it is recommended to use `vec_with_capacity()` whenever possible.
63 // One can also use `vec_resize()` or `vec_reserve()` before pushing many values
64 // onto the vector, which may force it to reallocate multiple times.
65 //
66 //
67 // # Safety
68 // The C language type system being weak and not enforcing any special rules
69 // for type conversion, the users of this library shall be especially careful
70 // with how they use the `vec_t` type.
71 //
72 // As its goal is to provide a way to store elements generically, it
73 // internally uses `void*` (which do not hold any information about the type
74 // being stored, but only where it points at in memory).
75 // The use of void pointers is generally dicouraged and considered bad practice
76 // as it often leads to many memory problems and UB.
77 //
78 // Therefore, the user must abide to these (non-exhaustive) rules:
79 // - Never manually change the fields of the vector.
80 // - Always use the vector with the type it was declared with.
81 // - Verify that the functions return valid pointers, values or the VEC_OK macro.
82 //
83 // Not following these rules WILL break your code and/or make you go insane
84 // while trying to track down the bugs! (Aslo, feel free to suggest/add rules in
85 // the list above).
86 //
87 //
88 // # Guarantees
89 // The `vec_t` type defined here is quite fundamental in nature as it only consists
90 // in a (length, capacity, element size, pointer) quadruplet. These are not
91 // necessarily stored together and their order is not specified. As mentioned in
92 // the usage rules defined the Safety section above, use the appropriate functions
93 // to modidy these.
94 // By design, `vec_t` is as low-overhead as the C language permits it, while trying
95 // to keep it as safe to use as possible.
96 //
97 // If a `vec_t` has allocated memory, then the memory it points to is on the heap.
98 // Its pointer points to `len` initialized, contiguous elements in order, followed
99 // by `capacity - len` logically uninitialized, contiguous elements (see diagram
100 // above for a better visualization).
101 //
102 // The `vec_t` type will never automatically shrink itself, even if completely empty.
103 // This ensures no unnecessary allocations or deallocations occur. Emptying a vector
104 // and then filling it back up to the same len should not incur calls to allocator
105 // functions. If you wish to free up unused memory, use `vec_shrink_to_fit()`.
106 //
107 // The library's functions `vec_push()` and `vec_insert()` will never (re)allocate
108 // if the vector's capacity is sufficient. These will only (re)allocate if
109 // `len == capacity`. The vector's capacity is completely accurate, and can be
110 // relied on, provided that the user never manually modifies the vector's
111 // `capacity` field.
112 //
113 // The `vec_t` type does not guarantee any particular growth strategy when
114 // reallocating, nor when `vec_reserve()` is called. The current strategy is
115 // basically to only allocate space for one more element.
116 // One can modify the `VEC_GROWTH_FACTOR` macro down in the Macro section if it
117 // wishes to optimize reallocations.
118 // Whatever strategy is used will of course guarantee O(1) amortized push.

179 // Mutates the element at the specified index, assigning it `value`.
180 //
181 // # Safety
182 // - The caller must guarantee that the specified type corresponds to the
183 // type of the specified value. Both the type and the value must be the
184 // same size as the vector `elem_size`.
185 //
186 // # Failures
187 // - Returns a `VEC_ERR` if `self` is not a valid pointer.
188 // - Returns a `VEC_ERR` if the underlying data of the vector is not
189 // a valid pointer.
190 //
191 // # Panic
192 // - Stops the program if the specified index is equal to or greater than
193 // the length of the vector.
194 #define VEC_MUT(vec, type, index, value) \
195 { \
196  if (!vec || !vec->data) { \
197  return VEC_ERR; \
198  } \
199  \
200  if (index >= vec->len || index < 0) { \
201  printf("Error: index out of bounds, `len` is %lu but `index` is %u\n", \
202  vec->len, \
203  index \
204  ); \
205  vec_drop(vec); \
206  exit(-1); \
207  } \
208  \
209  ((type*)vec->data)[index] = value; \
210 }
